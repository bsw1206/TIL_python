# Git remote에 대한 보충사항

- git remote add origin remote_repo_url에서 주소 대신 origin이라는 이름으로 간편하게 사용하기 위해 정하는 것
- local 이름은 origin 이외에도 아무거나 쓰일 수 있음.
- push : (***변경사항***)만큼만 업로드(예를 들어서 이미 commit 1, commit 2가 있었을 경우 commit 3이 새로 생성되었을 때, commit 3만 업로드)
- pull도 마찬가지로 변경사항만큼만 받음

## -> GitHub 사이트에서 유의할 점
> - 새로운 repository를 제작할 때 readme를 on 한 상태로 제작하면 이미 readme.md라는 것을 생성하는 commit이 되어 push가 안되기 때문에, clone을 통해 readme.md를 받고 다시 push를 해야 함!


> - git credential : 원격 저장소 입장에서 본인에게 push하는 사용자가 원격저장소에 권한이 있는 사람인지 확인하는 절차(아무나 push를 하면 안되기 때문에) -> 컴퓨터에 1개만 저장됨./컴퓨터의 "자격 증명 메뉴"에 저장됨. -> 자리를 옮기면 바꿔줘야 함. 아니면 새로 계정을 등록해야 함.
> - git config : 커밋에 작성되는 서명일 뿐(인증수단 X)
> - gitignore : git이 존재를 모르도록 설정(예시 숙지!)
> - gitignore.io : git으로부터 무시되어야 할 파일 이름들이 나열되어 있음.

# 확인 문제 풀기
1. 원격 저장소는 개인 컴퓨터 내에서만 버전 관리가 이루어지지는 않고, git 초기화시에 로컬 환경에 자동으로 생성된다. 버전 관리 이력도 보여주며, 온라인상에서 코드와 버전 관리 이력을 저장하는 공간이다. 답 : b
2. 로컬 저장소에 원격 저장소를 추가하기 위해 사용하는 명령어는 git push add <별칭> url 이다 답 : a
3. 로컬 저장소의 commit 목록을 원격 저장소로 업로드할 때 사용하는 명령어는 git push이다 답 : b
4. 원격 저장소 전체를 내 컴퓨터로 처음 가져올 때 사용하는 명령어는 git clone이다. 답 : d
5. .gitignore는 이름 앞에 마침표를 찍어야 하고, git 추적에서 제외하기 위해 사용되며, 이미 commit이 되면 무시가 안되기 때문에 주의하라고 하셨다. 답 : a
6. git revert는 commit을 완전히 삭제하는 것이 아니다. 특정 commit의 변경 사항을 반전시킨 새 commit을 만든다. 답 : d
7. git push는 별도의 인증 과정 없이 동작하지 않는다. 원격 저장소에는 commit이 올라간다. commit 이력이 업어도 사용 가능하며, 원격 저장소로 commit을 업로드 하는 기능이다. 답 : b
8. git reset을 사용 할때 commit에 포함된 기록을 남기지 않는 옵션은 --hard이다.
--mixed는 변경된 기록까지 삭제하는 것이고, --sort는 commit만 삭제하는 것이다. 답 : b
9. staging area에 추가된 파일을 변경사항을 유지하고 working directory로 되돌리는 용어는 git restore --staged <filename>이다.
10. commit 이력을 push하면서 master 브랜치의 원본을 기록하여 이후 원격 브랜치를 생략할 수 있도록 해주는 명령어는 git push --set-upstream origin master이다.

# Django Template system
- python data(context) + html 문서(template) -> 로직과 표현을 분리하여 동적인 웹 페이지 생성

### html의 콘텐츠를 변수 값에 따라 변경하기(실습)
- render(request, template 파일 위치, context를 지정 가능)
- template위치 : 해당 html에서 활용할 data를 context(3번째 위치)로 넘기는 곳이고 반드시 dictionary 형태로 넘겨주어야 함.
- views.py에서 dictionary 형태의 key 값을 index.html에서 입력하는데 중괄호 2개를 붙여주어야 함.

# Django Template Language(DTL)
- template에서 조건, 반복, 변수 등의 프로그래밍적 기능을 제공하는 시스템

# 변수, 필터, 태그, 커맨드(변수와 태그 사용이 중요함!)
- 변수 : render 함수의 세번째 인자, 해당 딕셔너리의 key값이 template에서 사용 가능한 변수명이 된다.
- 변수는 dot('.')을 사용하여 변수 속성에 접근할 수 있다. context = {'attribute : 'value_2',} -> {{variable_2.attribute}}
- 필터 : 표시할 변수를 수정할 때(변수 + '|' + 필터명)
- {{variable|filter}} 의 예시에서 공백이 있으면 안됨.(ex) {{name|truncatewords:30}}
- 태그 : 반복과 논리를 수행하여 제어 흐름을 만든다 {% 태그명 %} / 일부 태그는 시작과 종료 태그가 존재한다.
- 예시 : {% tag %}와 같이 단일 태그도 존재하고 범위가 필요할 때는 쌍을 가지는 태그도 가능하다. {% if %} {% endif %}
- views.py에서 범위를 저장하고 html에서 값을 활용한다는 의미로 받아들이는 것(이미 python에서 완성된 값을 html에 넣는다.)
- comments(커맨드)
- 주석 {# name #} 등으로 표현도 되지만 {% comment %} {% endcomment %}

# 템플릿 상속
- 페이지의 공통 요소를 포함, 하위 템플릿이 재정의 할 수 있는 공간을 정의
- 예를 들어서 어떤 홈페이지에서 윗 부분은 안 바뀌는 데, 아랫 부분의 내용만 계속해서 바뀜.
- <h1>네비게이션 바</h1> {% block content %} {% endblock content %} : 공통 요소 / 다른 html파일에서 {% extends "articles/base.html" %}     {% block content %} {% endblock content %}이 존재 : extends "articles/base.html" 이라는 것은 base.html파일을 확장해서 다른 html파일에서 사용하겠다는 의미이다. (부모 템플릿이랑 유사하다고 생각하면 됨. / 2개 이상 사용하면 안댐.)




# 데이터를 보내고 가져오기
- htmle form 요소를 통해서 사용자와 어플 간의 상호작용 이해
- 사용자의 입력을 받는 과정에서 form 태그는 필수이다.
- name 속성 : 사용자의 입력을 저장하는 속성 (ex : name='query' / 사용자의 입력을 쿼리에 저장하는 형태) -> 반드시 같이 입력해주어야 입력값이 저장된다!
- 예시 : https://search.naver.com/search.naver?query=hello에서 'https://search.naver.com/search.naver'는 목적지 url(보낼 서버 주소)이고,<input type="text" name="query" id="message">를 통해서 내 input 메세지를 query에 저장한 형태로 네이버로 전송함.
- action & method : 어디(action)로 어떤 방식(method)으로 전송할지


### query string pararmeters
- http://host:port/path?key=value&key=value
- 사용자의 입력 데이터를 URL주소에 파라미터를 통해 서버로 보내는 방법
- 문자열은 앰퍼샌드('&')로 연결된 key=value쌍으로 구성되며, 기존 URL과는 물음표('?')로 구분됨.

# URL 
- URL dispatcher : URL 패턴 정의, 해당 패턴이 일치하는 요청을 처리한 view 함수 연결(mapping)
- variable routing(중요!!) : URL 일부에 변수를 포함시키는 것(변수는 view 함수의 인자로 전달 가능)
- <path_converter:variable_name>
- urlpatterns = [path('articles/<int:num>/', views.detail),] -> 이렇게 선언할 때 views.py에서 def detail(request, num)와 같이 함수에 인자로 정의를 해주어야 함. 이름도 반드시 같아야 함!!
- App URL mapping : 각 앱에 URL을 정의하는 것(각 앱에 urls.py를 제작하는 ) -> 프로젝트와 각 앱이 URL을 나누어 관리를 편하게 하기 위함.



```python
# views.py

def detail(request, num):
    context = {
        'num': num,
    }
    return render(request, 'articles/detail.html', context)



# <!-- articles/detail.html>
{% extends "articles/base.html" %}
{% block content %}
  <h1>Detail</h1>
  <h2>{{ num }}번 게시글입니다.</h2>
{% endblock content %}

```

### url tag
- 주어진 URL 패턴의 이름과 일치하는 절대 경로 주소를 반환 {% url 'url_name' arg1 arg2 %}
- 예시 : <a href="{% url, 'detail' 1 %}">Article 1</a> 에서 'detail'다음에 1을 빠뜨리는 경우가 많은데 포함시키는 것이 매우 중요함!



###### app_name 속성
- app_name = 'articles' or app_name = 'pages' -> 같은 이름이더라도 pages의 index인지 articles의 index인지 지정 가능
